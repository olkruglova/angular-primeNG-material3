Dependency Injection — all cases

1. providedIn: 'root' + inject()

Interview: Registers the service in the root injector — one singleton for the whole app. Tree-shakeable: if nothing injects it, Angular removes it from the bundle. inject() is the modern API (Angular 14+); works in components, guards, resolvers, and plain functions — no constructor needed.

Instance ID (same everywhere in the app): {{ rootLogger.instanceId }}

2. Constructor injection

Interview: The classic Angular DI syntax. Angular reads the constructor parameter types at runtime and resolves them. Still valid — useful when you want dependencies to be explicit and visible at a glance. inject() is preferred for new code.

RemoteConfigService injected via constructor: {{ remoteConfig.constructor.name }}

3. Component-scoped provider

Interview: List the service in the component's providers: []. Each component instance gets its own service instance, isolated from siblings. Destroyed when the component is destroyed. Use for per-form, per-modal, per-widget state.

4. InjectionToken + useValue

Interview: Inject non-class values (config objects, feature flags, API URLs) in a type-safe way. The generic on InjectionToken<T> ensures the consumer gets the correct type. No class needed — any value works.

apiUrl: {{ config.apiUrl }}

darkMode: {{ config.featureFlags.darkMode }}

analytics: {{ config.featureFlags.analytics }}

5. useClass — swap implementations

Interview: Provide a different class for a token without changing any consumer. Angular creates a NEW instance of the replacement class. Common uses: swap real service for a mock in tests, or swap implementations based on environment / feature flag.

Parent provides: {{ "{" }} provide: AbstractLogger, useClass: ConsoleLoggerService {{ "}" }}

Child overrides: {{ "{" }} provide: AbstractLogger, useClass: SilentLoggerService {{ "}" }}

6. useFactory

Interview: Use when the provided value depends on other injected services or runtime conditions. The factory function receives the resolved deps as arguments. Angular calls the factory once and caches the result (within the injector scope).

GreeterService built by factory using AppConfig.apiUrl:

{{ greeter.greet() }}

7. useExisting — alias / interface pattern

Interview: Makes two tokens point to the same existing instance — no new object is created. Classic use: components depend on an abstract class token; the concrete service is registered under both. Unlike useClass, which creates a fresh instance, useExisting reuses whatever is already in the injector.

8. multi: true

Interview: Multiple providers contribute values to a single token — Angular collects them all into an array. This is how Angular's own HTTP_INTERCEPTORS and APP_INITIALIZER work internally. Each { provide: TOKEN, useValue: ..., multi: true } appends one entry.

Registered handlers ({{ logHandlers.length }}):

    @for (h of logHandlers; track h.name) {
  • {{ h.name }}
    • }

9. @Optional / inject(T, { optional: true })

Interview: If no provider exists for the token, Angular normally throws. optional: true returns null instead — graceful degradation. Use for optional features: analytics, crash reporting, feature toggles that may not be configured in all environments.

10. @Self / @SkipSelf

Interview: Control which injector in the tree Angular searches. @Self — only look in my own injector (throw if not there). @SkipSelf — skip my injector, start from the parent. Common in directives/services that need to distinguish their own instance from an ancestor's.

11. APP_INITIALIZER

Interview: A multi-provider token. Each factory function it registers runs before Angular renders the app. If it returns a Promise or Observable, Angular waits for it to complete. Use for: loading remote config, verifying auth tokens, preloading lookup data. Registered in app.config.ts.
